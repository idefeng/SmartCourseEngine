# SmartCourseEngine 重构计划

## 一、当前项目分析

### 1.1 现有架构
- **项目类型**: 智能课件生成引擎
- **核心功能**: 文档解析 → 知识提取 → 课件生成 → 视频生成
- **技术栈**: Python + Streamlit + ChromaDB + HeyGen API
- **架构模式**: 单体应用，模块化设计

### 1.2 优势
- ✅ 完整的课件生成流水线
- ✅ 数字人视频生成功能
- ✅ 知识库管理（ChromaDB）
- ✅ Streamlit可视化界面
- ✅ 容器化支持（Docker）

### 1.3 不足（相对于AI课程知识库需求）
- ❌ 缺乏视频内容分析能力
- ❌ 缺乏多模态AI分析（语音+视觉）
- ❌ 缺乏知识图谱构建
- ❌ 缺乏精准时间戳定位
- ❌ 缺乏动态知识更新机制
- ❌ 架构不够可扩展

## 二、重构目标

基于AI课程知识库技术方案，将SmartCourseEngine重构为：

### 2.1 新定位
从"智能课件生成引擎"升级为"**AI驱动的智能课程知识库与生成平台**"

### 2.2 核心能力扩展
1. **视频智能分析**: 支持现有课程视频的知识点提取和标签分类
2. **多模态知识库**: 构建包含文本、语音、视觉的多模态知识库
3. **智能检索系统**: 支持知识点级别的视频检索和时间戳定位
4. **动态知识更新**: 支持新内容（如法律法规）的录入和已有课程更新
5. **个性化学习**: 基于知识图谱的个性化学习路径推荐

### 3.2 微服务划分
1. **api-gateway**: API网关服务
2. **course-generator**: 课件生成服务（保留原有功能）
3. **video-analyzer**: 视频分析服务（新增）
4. **knowledge-base**: 知识库管理服务（升级）
5. **search-engine**: 智能检索服务（新增）
6. **recommendation**: 推荐服务（新增）
7. **notification**: 通知服务（新增）

## 四、重构步骤

### 阶段一：基础架构升级（2-3周）
1. **数据库迁移**
   - 从ChromaDB迁移到PostgreSQL + Pinecone + Neo4j
   - 设计新的数据模型
   - 数据迁移脚本

2. **API架构重构**
   - 从Streamlit单体应用拆分为微服务
   - 实现FastAPI网关
   - 统一认证和权限管理

3. **容器化升级**
   - 更新Docker配置支持多服务
   - 添加Kubernetes部署配置
   - 配置服务发现和负载均衡

### 阶段二：核心功能增强（3-4周）
1. **视频分析服务开发**
   - 集成Whisper语音识别
   - 集成CLIP/YOLO视觉分析
   - 实现知识点提取算法
   - 实现时间戳关联

2. **知识库服务升级**
   - 实现知识图谱构建
   - 实现多模态向量存储
   - 实现动态更新机制
   - 实现版本管理

3. **检索服务开发**
   - 实现多模态混合检索
   - 实现时间戳精确定位
   - 实现个性化排序

### 阶段三：功能集成与优化（2-3周）
1. **原有功能迁移**
   - 课件生成功能迁移到新架构
   - 数字人视频生成功能保留
   - Streamlit管理后台升级

2. **系统集成**
   - 服务间通信和事件驱动
   - 统一日志和监控
   - 性能优化和缓存策略

3. **测试和部署**
   - 单元测试和集成测试
   - 性能测试和压力测试
   - 生产环境部署

## 五、技术选型更新

### 5.1 新增技术栈
- **后端框架**: FastAPI（微服务），Gin（高性能服务）
- **数据库**: PostgreSQL（主数据），Neo4j（知识图谱），Pinecone（向量搜索）
- **消息队列**: RabbitMQ/Kafka（事件驱动）
- **AI模型**: Whisper（语音识别），CLIP（视觉分析），BERT（文本理解）
- **监控**: Prometheus + Grafana + ELK Stack

### 5.2 保留技术栈
- **前端**: Streamlit（管理后台）
- **视频生成**: HeyGen API（数字人视频）
- **容器化**: Docker + Docker Compose
- **开发语言**: Python 3.10+

## 六、数据模型设计

### 6.1 核心实体
```python
class Course:
    id: str
    title: str
    description: str
    video_url: str
    duration: int
    created_at: datetime
    updated_at: datetime

class KnowledgePoint:
    id: str
    course_id: str
    name: str
    description: str
    category: str
    importance: int
    start_time: int  # 时间戳（秒）
    end_time: int    # 时间戳（秒）
    embedding: List[float]  # 向量嵌入

class Tag:
    id: str
    name: str
    type: str  # 'category', 'topic', 'skill', 'difficulty'

class User:
    id: str
    name: str
    learning_history: List[LearningRecord]
    preferences: Dict[str, Any]
```

### 6.2 关系设计
- Course --(contains)--> KnowledgePoint
- KnowledgePoint --(related_to)--> KnowledgePoint
- Course --(has)--> Tag
- KnowledgePoint --(has)--> Tag
- User --(studied)--> KnowledgePoint

## 七、API设计

### 7.1 核心API端点
```
# 视频分析API
POST /api/v1/videos/analyze          # 分析视频内容
GET  /api/v1/videos/{id}/analysis    # 获取分析结果

# 知识库API
POST /api/v1/knowledge/points        # 创建知识点
GET  /api/v1/knowledge/search        # 搜索知识点
PUT  /api/v1/knowledge/{id}          # 更新知识点

# 检索API
GET  /api/v1/search                  # 综合搜索
GET  /api/v1/search/temporal         # 时间戳搜索

# 课件生成API（保留）
POST /api/v1/courses/generate        # 生成课件
POST /api/v1/videos/generate         # 生成数字人视频
```

## 八、迁移策略

### 8.1 数据迁移
1. **导出现有数据**: 从ChromaDB导出向量和元数据
2. **数据转换**: 转换为新的数据模型
3. **批量导入**: 导入到新数据库
4. **验证一致性**: 确保数据完整性和一致性

### 8.2 功能迁移
1. **并行运行**: 新旧系统并行运行一段时间
2. **功能逐步迁移**: 按模块逐步迁移功能
3. **流量切换**: 逐步将流量切换到新系统
4. **旧系统下线**: 确认新系统稳定后下线旧系统

### 8.3 用户迁移
1. **透明迁移**: 用户无感知迁移
2. **数据同步**: 确保用户数据完整迁移
3. **功能兼容**: 确保原有功能在新系统中可用
4. **用户通知**: 提前通知用户系统升级
